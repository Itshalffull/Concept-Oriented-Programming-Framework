# Composable formal verification for concept-oriented software

**Formal verification is decomposing into composable, spec-driven units that align naturally with Clef's concept architecture.** The convergence of three trends — LLMs achieving near-saturation on formal proof benchmarks (99.6% on MiniF2F), industrial adoption of contract-based compositional verification (AWS, Meta, Microsoft), and the emergence of stochastic contract frameworks for AI-integrated systems — creates an unprecedented opportunity to design verification as a first-class concept suite. The key architectural insight: Clef's independent concepts with relational state and declarative syncs map directly onto assume-guarantee reasoning, where each concept's operational principle becomes a verifiable temporal property and each sync becomes a contract composition point. This report synthesizes findings across tool architectures, LLM-assisted proving, industrial pipelines, compositional methods, and AI system verification to inform the design of Clef's verification concepts.

## How verification tools decompose their domains

Every major formal verification tool decomposes its problem space into a layered pipeline: **specifications** (what should hold) decompose into **properties** (individual claims) which generate **proof obligations** (formulas to prove), dispatched to **solvers** (decision procedures). The critical architectural pattern is what Lamport formalized in TLA+: **composition is conjunction**. Composing two verified specifications means taking their logical conjunction — if each component satisfies its local properties, their conjunction holds for the composition.

The tool-specific decomposition models reveal distinct but complementary strategies. **TLA+** uses modules with `EXTENDS`/`INSTANCE` for parameterized composition and refinement mappings for abstraction layers — a concrete spec Ψ refines abstract spec Φ if every behavior of Ψ is allowed by Φ. **Alloy** models everything as relations (sets are unary relations, scalars are singletons), checking all instances within a bounded scope via SAT solving. Alloy 6 adds temporal operators (`always`, `eventually`, `until`) over mutable state marked with `var`, enabling direct specification of safety and liveness properties. **Dafny** implements a three-layer pipeline (Dafny → Boogie intermediate verification language → Z3 SMT solver) where each method is verified independently against its contract — callee bodies are never inlined, which is the key to scalability. **Event-B** generates proof obligations per event (invariant preservation, guard strengthening, action simulation) and supports layered refinement through data refinement, event splitting, and new event introduction.

For concurrent systems, **CSP** provides algebraic composition operators (parallel `∥`, choice `□`/`⊓`, hiding `\`, sequential `;`) with well-defined algebraic laws. FDR checks trace/failures/failures-divergences refinement between process specifications. Temporal logics split into **LTL** (linear-time, properties over individual paths — model checking is PSPACE-complete) and **CTL** (branching-time, properties over computation trees — model checking is linear in state space × formula size). Every temporal property decomposes into a safety component ("nothing bad happens") and a liveness component ("something good eventually happens").

The solver layer provides the computational foundation. **Z3** and **cvc5** decide satisfiability modulo theories (linear arithmetic, bit-vectors, arrays, strings) via the SMT-LIB standard interface. The integration pattern is universal: Dafny → Boogie → Z3, Alloy → Kodkod → SAT, CBMC → SSA → SAT/SMT, Certora → EVM bytecode → SMT. Quantifier handling remains the key challenge — tools use E-matching/triggers or model-based quantifier instantiation, or restrict to decidable fragments.

For Clef's target of Solidity, **Certora Prover** decomposes verification through CVL (Certora Verification Language) into rules (expected behavior of state transitions), invariants (state properties maintained across all public functions), ghost variables (specification-only state for aggregate properties), and hooks (tying ghost state to storage operations). The **K Framework/KEVM** takes a semantics-first approach, defining complete formal semantics of the EVM and deriving verifiers automatically using reachability logic. The **Move Prover**, co-designed with its language, introduces specification schemas — named, reusable specification fragments applied via `apply` mechanism — and benefits enormously from an **alias-free memory model** that eliminates the frame problem, directly paralleling Clef's independent concept state.

## LLMs are transforming formal proof generation

The field has undergone explosive progress. **MiniF2F-test**, the standard benchmark for formal mathematical reasoning in Lean, has gone from ~29% (GPT-f, 2020) to **99.6%** (Seed-Prover, ByteDance, 2025). PutnamBench, the new frontier of undergraduate competition-level problems, reached **70.0%** (462/660 problems) with Apple's Hilbert system. This progress was driven by the insight that formal verification provides perfect reward signals for reinforcement learning — the proof either type-checks or it doesn't.

The dominant paradigm combines **recursive decomposition** with **informal-formal reasoning fusion**. **DeepSeek-Prover-V2** (671B parameters, April 2025) uses a large model to decompose problems into subgoals, a smaller 7B model to prove each subgoal, and synthesizes chain-of-thought with formal proofs for RL cold-start, achieving **88.9%** on MiniF2F. **Hilbert** (Apple, NeurIPS MATH-AI 2025) combines informal reasoning LLMs (Gemini 2.5) with specialized prover LLMs and a semantic theorem retriever in an agentic framework with recursive subgoal decomposition. **AlphaProof** (DeepMind, Nature November 2025) solved 3 of 5 non-geometry IMO 2024 problems using RL-based Lean proving, scoring **28/42 points** (silver medal).

For practical software verification, LLM-based **invariant generation** has become production-ready. Reasoning-optimized LLMs (O1, O3-mini) with Z3 verification loops achieved **100% coverage** on the Code2Inv benchmark (133 programs) in 1-5 iterations. **NeuroInv** combines LLMs with Hoare logic backward-chaining for **99.5%** success on 150 Java programs. **LaM4Inv** uses a "query-filter-reassemble" strategy — LLM generates candidates, bounded model checking filters valid predicates, SMT solver assembles new candidates — solving **309/316** benchmark programs. For smart contracts specifically, **PropertyGPT** (NDSS 2025) uses retrieval-augmented generation to produce formal properties verified by Certora, while **FLAMES** fine-tunes LLMs for invariant injection into Solidity contracts with a **96.7%** compile rate.

Specification generation from natural language is emerging but less mature. Research shows LLMs can write Alloy formulas from natural language descriptions, create equivalent alternative formulations, and complete specification sketches. **LMGPA** applies LLM-guided hierarchical decomposition to TLA+ proof obligations. **Laurel** achieves **56.6%** assertion generation rate for Dafny with GPT-4o. The critical gap remains **semantic validation** — correct syntax doesn't guarantee the specification captures the intended meaning. Mutation testing and LLM-generated test cases (using GPT-5 to generate positive/negative test cases for Alloy models) are emerging validation techniques.

LLMs and traditional automated theorem provers have **complementary strengths**. Their union consistently outperforms either alone — Baldur (whole-proof LLM generation) combined with Thor (Sledgehammer-based) reaches **65.7%** on Isabelle theorems versus ~47-57% individually. LLMs excel at generating high-level proof strategies and handling novel contexts; ATPs excel at routine symbolic manipulation and decidable theories. The open-source ecosystem is accelerating: LeanDojo, ReProver, Lean Copilot, and DeepSeek-Prover-V2 are all MIT-licensed.

## Industrial verification pipelines show the path to practice

Three industrial deployments illustrate how formal verification scales. **AWS** uses TLA+ for design verification of DynamoDB, S3, and EBS (engineers learn TLA+ in 2-3 weeks), Zelkova for IAM policy verification (scaling to **1 billion SMT queries/day**), and s2n-bignum for formally verified cryptographic assembly via HOL Light. **Meta's Infer** uses separation logic with **bi-abduction** — automatically inferring both missing preconditions and preserved frames for each function — enabling procedure-level independent analysis without human annotations. Infer processes thousands of code changes daily, identifying **1,000+ bugs per month** fixed before production, running in differential mode that only re-analyzes changed files. **Microsoft's Boogie** serves as an intermediate verification language connecting multiple front-ends (Dafny, VCC, Spec#, SMACK) to Z3, with Verus (Rust verification) achieving two of three best papers at OSDI 2024.

The integration patterns that matter for Clef are:

- **Intermediate verification languages** (Boogie, Why3, SIL) as composition points between source languages and solvers, enabling tool reuse
- **Incremental/modular verification** as the key to CI integration — verify only changed components and their dependents
- **Tiered verification** with fast lightweight checks on every commit and heavier proofs nightly or on merge
- **Frame specifications** (`modifies`/`assigns`/`reads` clauses) that explicitly declare what state an action can change, with everything else automatically preserved

The **Project Everest** pipeline exemplifies continuous verification: F* code is verified nightly, compiled to C via KaRaMeL, and the generated verified crypto code runs in Firefox, the Linux kernel, and WireGuard. The key lesson is that **verification must be as automated and incremental as compilation** to survive in production workflows.

The spectrum from testing to full proofs is not a binary choice but a continuum. Property-based testing (QuickCheck/Hypothesis) checks universally quantified properties with random inputs. Bounded model checking (CBMC, Alloy) exhaustively checks all inputs within a bound. Full verification (Lean, Dafny) provides mathematical proof. The composable unit is the **property specification itself** — it's tool-agnostic and can be checked at any level of the spectrum. **Concolic testing** (SAGE discovered ~1/3 of all security vulnerabilities in Windows 7 file fuzzing) and hybrid approaches (Driller, SymCC) combine fuzzing with symbolic execution, achieving coverage neither approach reaches alone.

## Compositional verification through contracts and assume-guarantee reasoning

Assume-guarantee (AG) reasoning is the foundational technique for compositional verification. It decomposes verification of a composed system M1 ∥ M2 into smaller subproblems: prove that M1 satisfies property A (an assumption about M2), then prove that M2 satisfies the global property P assuming A. This avoids state-space explosion from composing components. **Learning-based approaches** using Angluin's L* algorithm automatically discover the weakest safe assumption as a regular language, eliminating the need for manual assumption engineering. **AGREE** (Assume Guarantee REasoning Environment) implements this for aerospace systems using k-induction over AADL architectural models.

Contract-based design, pioneered by Bertrand Meyer's Design by Contract in Eiffel, provides the practical interface. Preconditions (client's obligation), postconditions (supplier's obligation), and class invariants compose through well-defined rules: conjunction strengthens guarantees, refinement weakens assumptions while strengthening guarantees, and compatibility checking ensures each component's guarantees satisfy the other's assumptions. **Ada SPARK** combines contract-based flow analysis with SMT-based proof via GNATprove/Why3, deployed in NVIDIA firmware and Thales avionics. **JML** for Java provides quantifiers, old-value references, result references, and frame conditions. The **Move Prover** introduced specification schemas — reusable specification fragments applied across multiple functions — directly relevant to Clef's pattern of composable concepts.

**Interface automata** (de Alfaro & Henzinger, 2001) formalize temporal aspects of component interfaces using an optimistic composition model — two components are compatible if *some* environment makes them work together. **Multiparty session types** extend this to N-party protocols, where a global type describes the overall protocol and projection derives each participant's local type. Well-typed processes are guaranteed **deadlock-free** and **communication-safe** — properties preserved through execution via subject reduction.

**Separation logic** provides the deepest foundation for independent-state compositional reasoning. The **separating conjunction** (P ∗ Q) asserts two memory regions are disjoint, and the **frame rule** guarantees that verifying an action against a small footprint automatically preserves untouched state. **Iris** (implemented in Coq) extends this to higher-order concurrent separation logic with customizable resource algebras (cameras). For Clef, since concepts have independent state, the frame rule applies directly: verifying a concept's action requires reasoning only about that concept's state. Other concepts' state is automatically preserved.

## Verifying systems with LLM components requires probabilistic contracts

Traditional formal verification assumes deterministic transitions, but LLMs are inherently stochastic. The field is converging on **treating LLMs as untrusted oracles** and verifying the surrounding enforcement mechanisms. Three frameworks define the current frontier.

**Pro2Guard** (Wang et al., 2025) models LLM agent behavior as Discrete-Time Markov Chains learned from execution traces, then performs probabilistic model checking to predict if an agent is likely to reach an unsafe state. If the probability exceeds a threshold, interventions trigger (halting, user verification, or LLM self-reflection). This works with the **Storm** and **PRISM** probabilistic model checkers.

The **Agent Behavioral Contracts (ABC) framework** (February 2026) introduces the most rigorous compositional approach: **(p, δ, k)-satisfaction** means a contract holds with probability ≥ p, with recovery within k steps. It models behavioral drift as a continuous-time stochastic process with restorative contract enforcement forces, proving a **Stochastic Drift Bound Theorem** via Lyapunov stability analysis. Critically, it proves a **Compositionality Theorem** for multi-agent pipelines: interface compatibility + assumption discharge + governance consistency + recovery independence are sufficient conditions for safe composition. The framework includes **ContractSpec** (YAML-based DSL) and **AgentAssert** (runtime enforcement with sub-10ms overhead).

**VeriGuard** (2025) takes the wrapper-verification approach: it synthesizes a behavioral policy, formally verifies it using Hoare triples via the Nagini verifier, then deploys a lightweight runtime monitor that validates each agent action against the pre-verified policy. This achieves near-zero attack success rate across GPT-4.1, Claude-Sonnet-4, and Gemini models. **AgentSpec** (ICSE 2026) provides a domain-specific language for specifying and enforcing runtime constraints on LLM agents, with LLM-generated rules achieving **87-96%** detection of risky cases.

**Conformal prediction** bridges statistical guarantees and formal methods. Distribution-free and model-agnostic, CP provides finite-sample coverage guarantees for LLM outputs. Key advances include conditional conformal prediction (NeurIPS 2024) that adaptively adjusts guarantee strength, and conformal language modeling (ICLR 2024) that calibrates stopping/rejection rules for sampling LLM outputs. The fundamental limitation: all statistical guarantees assume exchangeability, which adversarial inputs violate.

The **4/δ convergence bound** (Dantas et al., December 2025) provides a landmark theoretical result: modeling the LLM-verifier interaction as a sequential absorbing Markov Chain, the system reaches a Verified state almost surely for any non-zero stage success probability δ, with expected iterations bounded by **E[n] ≤ 4/δ**. This means LLM+verifier loops are guaranteed to terminate — a critical property for any verification pipeline incorporating LLMs.

## Designing Clef's composable verification concepts

Daniel Jackson's concept-oriented design theory (MIT, "The Essence of Software," 2021) maps directly to verification decomposition. Each concept has a purpose (singularity principle), operational principle (archetypal scenario), relational state, and actions. Concepts compose conjunctively through synchronizations while preserving individual properties (integrity principle). The OP is not a complete specification — it's the behavioral intuition ("if you set an alarm and turn it on, it rings at the set time") while the state machine model is the complete formal spec. The OP acts as a **high-level temporal property** the formal model must satisfy. This is precisely the relationship between TLA+ specifications and their temporal properties, or between Alloy models and their assertions.

Based on this synthesis, ten architectural patterns emerge for Clef's verification concepts:

**Pattern 1 — Concept-as-verification-unit.** Each verification concern becomes a first-class concept following Jackson's singularity principle. An `Invariant` concept manages predicates, witnesses, and violations. A `Refinement` concept tracks abstract specs, concrete implementations, and simulation relations. A `Contract` concept holds assumption-guarantee pairs for compositional reasoning. A `Safety` concept manages bad-state specifications and reachability analysis. Each has independent state, explicit actions, and its own operational principle.

**Pattern 2 — Small-scope checking as default.** Following Alloy's approach, each concept is verified by exhaustive checking within bounded scopes. The relational state model enables SAT-based analysis. Scope parameters are part of the concept's configuration. The small scope hypothesis — that most bugs manifest in small instances — is well-supported empirically and especially effective for Clef's naturally bounded concept state.

**Pattern 3 — Contract-mediated composition through syncs.** Syncs become contract composition points. Each concept exposes preconditions on actions (what it assumes about inputs), postconditions (what it guarantees about state/outputs), and invariants (what always holds). The sync mechanism verifies that each concept's guarantee satisfies connected concepts' assumptions. AG proof rules establish that compositions preserve global properties.

**Pattern 4 — The frame rule for independent state.** Since Clef concepts have independent state, separation logic's frame rule applies directly. Verifying a concept's action requires reasoning only about that concept's state — other concepts' state is automatically preserved. This is the same insight that makes the Move Prover practical (alias-free memory model) and Meta's Infer scalable (bi-abduction with frames).

**Pattern 5 — Operational principles as temporal properties.** OPs encode as Alloy 6-style temporal assertions: `always (trigger implies eventually response)` for response properties, `always (not bad_state)` for safety, `always eventually (progress)` for liveness. Clef's existing concept OPs already express these patterns informally — formalization means encoding them in temporal logic and checking them against the concept's state machine.

**Pattern 6 — Specification schemas for reuse.** Following Move Prover's schema mechanism and Dwyer's property specification patterns, common verification patterns (absence, existence, response, precedence, bounded existence) can be packaged as reusable specification templates applied to concepts. Smart contract patterns (reentrancy guards, overflow checks, access control invariants) become domain-specific schemas for Solidity-targeting concepts.

**Pattern 7 — Ghost state for cross-concept properties.** Following Certora's ghost variables and Iris's ghost state, specification-only state tracks properties spanning multiple concept interactions. For example, "total token supply is conserved across transfers" requires ghost state tracking aggregate balances that no single concept owns.

**Pattern 8 — LLM-assisted verification pipeline.** An `LLMProver` concept orchestrates LLM-based invariant generation, specification mining from code, and proof search. Given that LLM+SMT loops achieve 100% on standard invariant benchmarks and the 4/δ bound guarantees convergence, this becomes a practical verification accelerator. The concept manages the generate-check-repair cycle with the SMT solver providing the oracle.

**Pattern 9 — Stochastic contracts for LLM-integrated concepts.** For Clef's 24 LLM integration concepts, the ABC framework's (p, δ, k)-satisfaction provides the contract model. Each LLM-involving concept specifies probabilistic guarantees (output quality holds with probability ≥ p, recovers within k steps), and the Compositionality Theorem ensures pipeline-level safety from component-level contracts.

**Pattern 10 — Two-level verification with proof obligation management.** A `ProofObligation` concept tracks what needs to be proved, an `Evidence` concept manages test results, proofs, and model-checking results, and a `Verdict` concept combines evidence to reach conclusions. Top-level verification uses Alloy-style small-scope checking of concept models. Bottom-level verification uses Certora/Move Prover-style deductive verification of Solidity output. The entire verification workflow is itself composed of concepts — tracked, versioned, and composable.

## Conclusion

The design of composable formal verification concepts for Clef benefits from a deep structural alignment: **concept-oriented programming and compositional formal verification solve the same decomposition problem from opposite directions.** Concepts decompose software into independent, spec-driven units; compositional verification decomposes correctness into local proof obligations that compose into global guarantees. The critical technical enablers are the frame rule (independent state means independent verification), assume-guarantee reasoning (contracts compose algebraically), and temporal property decomposition (safety ∧ liveness covers all properties).

The LLM dimension introduces both challenge and opportunity. For verification *of* LLM-integrated concepts, probabilistic contracts with (p, δ, k)-satisfaction and the treat-LLM-as-untrusted-oracle pattern provide principled foundations. For verification *using* LLMs, the field has reached practical maturity for invariant generation (100% on standard benchmarks), proof search (99.6% on miniF2F), and specification assistance — with the 4/δ convergence bound guaranteeing termination of LLM-verifier loops. The most impactful near-term design decision is positioning Clef's verification concepts to consume both traditional solver outputs and LLM-generated proof artifacts through a unified `Evidence` interface, enabling graceful escalation from property-based testing through bounded model checking to full formal proof — all within the same concept composition framework.
